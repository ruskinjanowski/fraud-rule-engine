# Working Plan

**Provisional.** Current best-guess sequencing, not a commitment — revisit at the start of each iteration and re-validate against [BRIEF.md](BRIEF.md). Steps 1–5 cover the brief's core; later steps are differentiators (cut from the tail under time pressure, never from the middle). Each step should leave the project runnable.

Detailed scope of each step is decided when we pick it up (via the **plan-feature** skill), not here. Nothing in the backlog below is a requirement — the brief defines done; accepted architectural items get an ADR in `docs/adr/`.

## Sequence

- [x] 1. Foundation: Spring Boot 4.1 / Java 25 project, multi-stage Dockerfile, docker-compose (app + Postgres + Kafka), Flyway wired, env-var config, README. (CI skeleton still outstanding — deferred to step 9 hardening.)
- [x] 2. Domain model + database migrations (TransactionEvent, FraudEvaluation, RuleResult) — schema per [ADR-0001](adr/0001-domain-model-and-persistence-schema.md); Flyway V1, JPA entities, repositories, Testcontainers round-trip test
- [x] 3. Ingestion path: **Kafka consumer (sole ingestion path)** — a thin adapter over `FraudEvaluationService`, validating each message and idempotent by event ID, with a dead-letter topic for un-processable messages (per [ADR-0004](adr/0004-kafka-ingestion-adapter.md)). The earlier `POST /api/transactions` write endpoint was removed: ingestion is event-driven and the brief asks only for retrieval via API (see 2026-06-16 below)
- [x] 4. Rule engine + initial deterministic rules — engine + Rule SPI + **8 rules** (amount threshold, velocity, cumulative amount, card testing, duplicate, odd hours, impossible travel, high-risk country) per [ADR-0003](adr/0003-customer-history-context-for-stateful-rules.md); per-rule enabled flags. Baseline rules (amount deviation, dormant account) + contract-widening rules (MCC, device) remain backlog
- [x] 5. Decision persistence + retrieval API — persistence + `GET /api/evaluations/{eventId}` (detail) + paginated `GET /api/evaluations` list with combinable customer / occurred-at time-range / flagged filters (per [ADR-0005](adr/0005-paginated-retrieval-query-api.md))  ← *brief's core complete; next milestone gate: run evaluate-against-brief*
- [~] 6. Rule lifecycle & management (if shadow-mode ideas are accepted) — **shadow seam landed**: `Rule.advisory()` (ADR-0006) lets a rule be evaluated & persisted but excluded from the flag; full ACTIVE/SHADOW/DISABLED lifecycle + management API still backlog
- [ ] 7. Hindsight/analytics endpoints (if accepted)
- [x] 8. ML/LLM advisory rule — **ML anomaly scorer landed**: Isolation Forest (Smile) in shadow mode per [ADR-0006](adr/0006-ml-anomaly-scorer-in-shadow-mode.md); an LLM variant could reuse the same advisory seam
- [~] 9. Hardening: observability (structured ECS logging + trace correlation, Prometheus metrics with domain meters, GitHub Actions CI) **done**; demo data done; README polish + final evaluate-against-brief outstanding

- 2026-06-12: **Step 3 done (Kafka primary path).** ADR-0004: a `@KafkaListener` adapter (`messaging` pkg) over the existing `FraudEvaluationService` — the second entry point the service seam was built for (ADR-0002). Decisions: messages are the same JSON shape as the REST body, consumed as **String** and parsed with the app's Jackson `ObjectMapper` (sidesteps Spring-Kafka's Jackson-2 `JsonDeserializer`, which lacks jsr310/`Instant` support under Boot 4's Jackson 3) and validated with the same Bean Validation constraints; at-least-once redelivery absorbed by the service's `eventId` idempotency; **poison messages (malformed JSON / validation failures) → `<topic>.DLT`** via a `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` (non-retryable, dead-lettered on first encounter). REST kept as the demo/curl companion. Verified: 53 tests green incl. a Testcontainers Kafka end-to-end (publish → evaluate → store → retrieve) + two DLT scenarios (malformed and validation-failure both dead-lettered, partition keeps moving). Gotchas: (a) TC's `apache/kafka` container crashes on boot under its listener wiring — switched the *test* broker to `ConfluentKafkaContainer`/`cp-kafka` (compose stack stays on apache/kafka); (b) defining a custom `KafkaTemplate` suppresses Boot's autoconfigured one (`@ConditionalOnMissingBean`); (c) the recoverer's default DLT name didn't match the configured topic — pinned an explicit `<topic>.DLT` destination resolver.
- 2026-06-12: **Kafka adapter hardened to idiomatic Spring Kafka 4** (post-review pass; ADR-0004 updated in place). Three changes: (a) replaced the String-consume-and-parse workaround with the framework chain — `ErrorHandlingDeserializer` wrapping the Jackson 3 **`JacksonJsonDeserializer`** (`Instant` works out of the box; the typed listener no longer touches an `ObjectMapper`), with the DLT recoverer using two templates (raw `byte[]` for undeserializable records, JSON for validation failures); (b) adopted Spring Kafka 4's **default DLT name `<topic>-dlt`** (the default suffix changed from 3.x's `.DLT` — verified in spring-kafka 4.1.0 bytecode) and dropped the custom destination resolver; (c) test broker switched `cp-kafka:7.8.0` → **`apache/kafka-native:3.9.1`** (Testcontainers' recommended image, same lineage as compose, sub-second startup) and compose bumped `apache/kafka:3.9.0` → `3.9.1` to match. Root cause of the original "TC apache/kafka crashes on boot" gotcha found: a **3.9.0 image bug** — `launch` passes `-Dlog4j.configuration=…` to the `setup` command, which mis-parses it and fails storage format with a bogus "advertised.listeners cannot use 0.0.0.0"; fixed in the 3.9.1 images. Verified: 53 tests green (Kafka e2e + both DLT scenarios) against the native broker.
